# 代理程式說明

本文檔說明市集中各代理程式的用途與使用情境。

---

## 概述

代理程式是改變 Claude 工作方式的行為塑造工具，而非知識擴充。當觸發條件符合時會自動派遣，無需手動呼叫。

---

## planner

**用途**：複雜功能和重構的專業規劃專家

**自動觸發時機**：
- 使用者請求新功能實作
- 規劃跨多個檔案的複雜重構
- 架構變更或遷移
- 任何需要 3 步以上的任務

**功能**：
- 以清晰的語言重述需求
- 分解為分階段的實作步驟
- 識別元件間的依賴關係
- 評估風險和潛在阻礙
- 估算複雜度（高/中/低）
- **在繼續前等待使用者確認**

**輸出格式**：包含階段、步驟、依賴關係、風險和測試策略的結構化計畫

**整合技能**：`brainstorming/software-brainstorming` 用於需求探索

**評估結果**：100% vs 50% 基準（無此代理時計畫缺少階段、依賴關係和測試計畫）

---

## tdd-guide

**用途**：執行測試優先方法論的測試驅動開發專家

**自動觸發時機**：
- 實作任何新功能
- 修復缺陷或 bug
- 重構現有程式碼
- 任何產出程式碼的任務

**功能**：
- 優先搭建介面和型別定義
- 撰寫失敗測試（RED 階段）
- 實作最小程式碼使測試通過（GREEN 階段）
- 在測試保持綠色狀態下重構（REFACTOR 階段）
- 驗證 80%+ 測試覆蓋率

**TDD 循環**：
```
RED -> GREEN -> REFACTOR -> REPEAT
```

**覆蓋率要求**：
- 所有程式碼最低 80%
- 關鍵業務邏輯 100%

**評估結果**：100% vs 0% 基準（無此代理時，Claude 預設先實作、無測試）

---

## analyzer

**用途**：如同科學家般隔離變數並評估結果的系統性分析專家

**自動觸發時機**：
- 分析實驗結果或基準測試數據
- 除錯效能衰退或非預期行為
- 比較兩個或多個方案、配置或實作
- 理解模型、系統或流程為何有特定行為
- 評估選項之間的取捨

**核心原則**：一次只變更一個變數
1. 識別所有可能影響結果的變數
2. 除了調查中的變數外，保持所有其他變數不變
3. 測量單一變更的結果
4. 記錄結果後再進行下一個變數

**輸出格式**：
```markdown
# 分析：[問題]
## 設置（基準、指標、變數）
## 結果（含差異值的表格）
## 關鍵發現
## 混淆因子與限制
## 建議
```

**分析類型**：
- 效能分析
- 除錯 / 根因分析
- 比較分析
- 實驗結果分析

**整合技能**：`brainstorming/scientific-brainstorming` 用於假設生成

**評估結果**：100% vs 75% 基準（確保嚴謹的實驗思維，避免過早下結論）

---

## 派遣規則

- 若任務符合多個代理，按順序使用：**planner**（規劃）-> **tdd-guide**（實作）-> **analyzer**（評估）
- 獨立子任務可平行派遣多個代理
- 代理可與腦力激盪技能結合：planner 使用 software-brainstorming，analyzer 使用 scientific-brainstorming

---

## 代理結構

所有代理以 Markdown 檔案定義於 `agents/`，含 YAML 前置資料：

```yaml
---
name: agent-name
description: 含自動觸發條件的簡要描述
tools: ["Read", "Grep", "Glob", "Bash", "Write", "Edit"]
model: opus
skills:
  - skill-name
---
```
